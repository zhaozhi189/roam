# Roam 技术可行性验证(M0)v0.1

> 起草日期:2026-05-12
> 关联:[03-技术架构.md](03-技术架构.md) 是 C1 级架构(分 4 层、组件选型);本文是 **M0 阶段必须打死的假设清单**,先于任何应用代码。
> 时间盒:**周末 2 个半天(约 8 小时)**,超时即触发回退方案,不无限投入。

---

## 一、目的与适用范围

### 1.1 为什么先做可行性

Roam 的核心价值链是「**3DGS 文件 → 浏览器 → 流畅漫游**」。其中任何一环不成立,产品就不成立。**M0 的唯一目标是用最低成本证明这条链路能跑通**。

### 1.2 M0 通过 ≠ MVP 完成

M0 通过 = 6 个关键假设全部 ✅,可以放心进入工程;**不等于** 写完产品代码。M0 阶段的产物是 **demo + 数据 + 结论**,不是可用产品。

### 1.3 与 03-技术架构.md 的分工

- `03-技术架构.md`:**回答"是什么"**(分 4 层、用什么组件)
- 本文:**回答"行不行"**(关键假设能否被打死,以什么为通过标准)

## 二、假设清单(总览)

| # | 假设 | 风险 | 失败成本 | 验证顺序 |
|---|---|---|---|---|
| **H1** | PlayCanvas + .sog 在桌面 Chrome ≥ 60fps | 🟢 低 | 整个项目否决 | ① 先验 |
| **H2** | Android 中端机 ≥ 24fps,LOD 兜底可用 | 🔴 高 | 退守 Web 桌面优先,Android 降级 | ② |
| **H3** | iOS Safari 起步可用(iPhone 11+) | 🟡 中 | iOS 标"实验性",不做承诺 | ③ |
| **H4** | SplatTransform 真实场景压到 ≤ 30MB | 🟡 中 | 单场景上限调高 / 多分片 | ④ |
| **H5** | R2 + 流式加载首屏 < 10s(4G 模拟) | 🟡 中 | 预加载或 CDN warmup | ⑤ |
| **H6** | Workers + Hono + D1 + R2 + OAuth 走通 | 🟢 低 | 退守 Pages Functions 简化 | ⑥ |

> **不验证**:云端训练(Phase 2)、多场景拼接(Phase 2)、PWA 安装(Phase 2)、LLM 任何用法(永久不在 MVP)。

## 三、详细验证矩阵

### H1 · 桌面 Chrome 流畅度(基线)

| 项 | 内容 |
|---|---|
| **完整假设** | PlayCanvas Engine 加载 PlayCanvas 官方 FPS 示例(<https://blog.playcanvas.com/turning-a-gaussian-splat-into-a-videogame>)的 .sog 场景,在桌面 Chrome 上稳定 ≥ 60fps |
| **测试设备** | MacBook(zhi 主力机)+ 桌面 Chrome 最新版 |
| **验证手段** | 1. clone 官方 demo repo<br/>2. `npm install && npm run dev`<br/>3. 打开 `chrome://gpu` 确认硬件加速<br/>4. 用 Chrome devtools Performance 面板录 30 秒 |
| **通过标准** | 平均 ≥ 60fps,p99 < 16.7ms,**无明显掉帧** |
| **预估工时** | 1 小时 |
| **失败兜底** | 如果连官方 demo 都跑不动,**直接放弃 PlayCanvas 选型**,改回评估 Three.js + gsplat.js;此为最坏情况,概率 < 5% |

### H2 · Android 中端机帧率(关键)

| 项 | 内容 |
|---|---|
| **完整假设** | 同一份 .sog,在 2-3 年前的 Android 中端机(zhi 备用机)上 ≥ 24fps;LOD 自适应可以把帧率从初始 < 24 拉到 ≥ 24 |
| **测试设备** | zhi 的 Android 备用机(具体型号 TBD,需先查 GPU 型号) |
| **验证手段** | 1. 在 H1 demo 上加 `stats.js` 显示实时 fps<br/>2. Chrome remote debugging 看 Performance<br/>3. 测试 3 个场景大小:5MB / 15MB / 30MB |
| **通过标准** | 30MB 场景 ≥ 18fps 启动,LOD 调整后稳定 ≥ 24fps |
| **预估工时** | 2 小时(含初次配置 remote debug) |
| **失败兜底** | 不达标但 ≥ 18fps:把 Android 标为"基础体验",单场景上限降到 15MB;< 18fps:**Android 降级为 Phase 2 目标**,MVP 仅承诺桌面 |

### H3 · iOS Safari(可选,not blocker)

| 项 | 内容 |
|---|---|
| **完整假设** | iPhone 11(2019 款,作为底线) Safari 能加载 15MB 场景并漫游 |
| **测试设备** | zhi 没有 iPhone → 借设备或用 BrowserStack 远程 |
| **验证手段** | 1. 部署到 Cloudflare Pages 临时域<br/>2. iOS Safari 访问<br/>3. 观察是否 OOM 崩溃,fps 是否 ≥ 24 |
| **通过标准** | 15MB 场景不崩,fps ≥ 24 |
| **预估工时** | 1 小时(主要消耗在借设备) |
| **失败兜底** | iOS 在落地页明确标"实验性,部分机型可能不可用";不做承诺 |

### H4 · SplatTransform 压缩比

| 项 | 内容 |
|---|---|
| **完整假设** | 用 KIRI Engine 扫一个真实房间(zhi 自己卧室),原始 .ply 经 `splat-transform` 压缩后产物 ≤ 30MB,视觉无明显劣化 |
| **测试场景** | zhi 卧室(中等复杂度,~10㎡,含家具 / 纹理) |
| **验证手段** | 1. KIRI Engine 扫描 → 导出 .ply<br/>2. `splat-transform input.ply -K output.sog`<br/>3. 对比原始 vs 压缩在桌面 demo 中的视觉<br/>4. 测试不同压缩参数(若有) |
| **通过标准** | 产物 ≤ 30MB **且** 漫游时视觉劣化在主观可接受范围(模糊 / 噪点 / 缺失可见) |
| **预估工时** | 2 小时(含扫描时间) |
| **失败兜底** | 压缩后 > 30MB:把单场景上限调到 50MB(影响 R2 配额);视觉劣化严重:换 KIRI 不同模式重扫,或评估 Polycam |

### H5 · R2 + 流式加载首屏耗时

| 项 | 内容 |
|---|---|
| **完整假设** | 把 H4 产物上传 R2,通过 Cloudflare Pages 引用,4G 网络模拟下从「点链接」到「可漫游」**< 10 秒** |
| **测试环境** | Chrome devtools Network 节流到 Fast 4G(4Mbps DL,3 Mbps UL) |
| **验证手段** | 1. 把 .sog 手动上传 R2<br/>2. Pages 部署最小 demo 加载该 R2 URL<br/>3. devtools 测 `DOMContentLoaded` + `splat ready event` |
| **通过标准** | 12MB 场景,4G 节流下 < 10s 可漫游;p95 < 15s |
| **预估工时** | 1 小时 |
| **失败兜底** | 慢于 15s:加 PlayCanvas 渐进 LOD(先加载低密度版,后台流式补齐);最坏:在 P1 落地页加显式 loading 进度条 |

### H6 · 上传链路打通(端到端)

| 项 | 内容 |
|---|---|
| **完整假设** | 写一个最小后端:GitHub OAuth 登录 → 上传 .sog 文件 → 写 R2 → 写 D1 → 返回公开 URL,**全链路 < 200 行代码** |
| **验证手段** | 1. `wrangler init` 创建 Workers 项目<br/>2. Hono 加 2 个路由:`POST /api/upload` 和 `GET /api/auth/github/callback`<br/>3. D1 建一张 `scenes` 表(uuid, owner, name, size)<br/>4. 用 curl 或 Postman 走通 |
| **通过标准** | 端到端 demo 跑通,文件落 R2,元数据落 D1,公开 URL 可访问 |
| **预估工时** | 2 小时(主要在 Cloudflare 控制台配置) |
| **失败兜底** | OAuth 配置卡住:先用 mock token 走完链路;D1 卡住:用 R2 元数据 JSON 替代;**Hono 卡住:换原生 Workers handler** |

## 四、验证顺序与依赖

```
H1(桌面基线) ─┬─→ H2(Android)
              └─→ H3(iOS)

H4(扫描+压缩) ──→ H5(R2+加载) ──→ H6(上传链路)
```

**关键路径**:H1 → H4 → H5。**H6 可与 H2/H3 并行**(不依赖 splat 流畅度结论)。

## 七、总验收标准

M0 通过 = 以下全部满足:

- [ ] H1 桌面 ≥ 60fps 通过
- [ ] H2 Android 中端机 ≥ 24fps 通过(或确认降级路径)
- [ ] H4 卧室场景压缩到 ≤ 30MB 且视觉可接受
- [ ] H5 12MB 场景 4G 模拟 < 10s 可漫游
- [ ] H6 上传链路端到端跑通,可手动用 curl 上传一个文件并通过返回 URL 浏览
- [ ] H3 iOS 通过或显式标记"实验性"

**M0 失败的判定**:任何一项 H1/H2/H4 失败 = 整体失败,回到方案评估;H3/H5/H6 失败 = 调整边界后继续推进。

## 八、不验证什么

| 项 | 理由 |
|---|---|
| ❌ 云端训练(.ply 直接进系统) | F-07 是 Phase 2,MVP 用户在本地自己跑 SplatTransform |
| ❌ PWA 安装体验 | F-11 Phase 2,MVP 是普通网页 |
| ❌ 多场景拼接 / 大场景流式 | Phase 2/3,MVP 单场景 ≤ 30MB |
| ❌ LLM 任何用法 | 永久不在 MVP,见 [04-上下文交接.md § 1.3](04-上下文交接.md) |
| ❌ 商业化 / 付费链路 | "至少前 6 个月不商业化" |
| ❌ 多人协作 / 实时同步 | Phase 3 |
| ❌ 千元机以下机型 | 边界明确,不投入精力 |
