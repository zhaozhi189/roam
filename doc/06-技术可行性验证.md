## 三、详细验证矩阵

### ✅ H1 · 桌面 Chrome 流畅度(已通过)

| 项 | 内容 |
|---|---|
| **完整假设** | PlayCanvas FPS 博客 .sog 场景在桌面 Chrome ≥ 60fps |
| **测试设备** | MacBook + Chrome 147.0.7727.102,Mac OS X 26.4.1 |
| **测试素材** | <https://playcanv.as/p/qxGSuzYq/>(PlayCanvas 公网托管 demo) |
| **实际结果** | GPU 全硬件加速(WebGL / WebGL2 / WebGPU / Compositing / Rasterization 全绿);主观漫游流畅,移动无问题 |
| **结论** | ✅ 通过(2026-05-12) |

### ✅ H2 · Magic7 Pro 满血基线(已通过)

| 项 | 内容 |
|---|---|
| **完整假设** | Magic7 Pro(骁龙 8 Gen4 + Adreno 830)跑同一 demo ≥ 60fps |
| **测试方式** | 手机 Chrome 直接打开公网 demo URL |
| **实际结果** | **比 MacBook 还流畅**,zhi 主观评价 |
| **结论** | ✅ 通过(2026-05-12) — Adreno 830 在 3DGS 场景上确实强 |

### 🟡 H7 · Android MediaProjection 视频录制

| 项 | 内容 |
|---|---|
| **完整假设** | Android WebView 套壳里,用 MediaProjection API 录全屏(包含 PlayCanvas WebView 渲染 + UI 操作),输出 H.264 MP4,**60 秒 1080p@30fps 文件 ≤ 25MB**(微信限制) |
| **风险点** | (1) MediaProjection 能否捕获 WebView 内 WebGL/GPU 渲染?(2) 码率控制能否压到 25MB 以内?(3) 录制过程中 PlayCanvas 帧率会不会拖累 |
| **验证手段** | 1. 建最小 Android WebView 套壳,加载本地 PlayCanvas demo<br/>2. 加 MediaProjection 录屏代码<br/>3. 录 60 秒漫游,测试文件大小、视觉质量、PlayCanvas 帧率是否下降 |
| **通过标准** | 1080p@30fps 60 秒 ≤ 25MB,视觉无明显劣化,录制中 PlayCanvas 帧率仍 ≥ 30fps |
| **预估工时** | 3h(包括建 Android 项目 + 跑 MediaProjection demo) |
| **失败兜底** | 录制中帧率掉太多 → 降到 720p / 24fps;文件太大 → 缩短录制时长上限到 30s;最坏 → 改用 canvas.captureStream + MediaRecorder 只录 PlayCanvas 画面不录 UI |

### 🟢 H8 · 微信 Share Intent

| 项 | 内容 |
|---|---|
| **完整假设** | Android `Intent.ACTION_SEND` 带视频文件 URI,微信会出现在系统分享面板,选「微信」→「发送给朋友」→ 选联系人 → 视频成功发送到对话框,**对方能正常播放** |
| **风险点** | 微信的 IntentFilter 是否接受任意 mime type 的视频?FileProvider 配置是否正确,微信能否读到 content:// URI? |
| **验证手段** | 1. WebView 套壳里加一个「分享」按钮<br/>2. 触发 `Intent.ACTION_SEND` 带 video/mp4 + FileProvider URI<br/>3. 实际从 zhi 微信发到自己,看对方收到画质 |
| **通过标准** | 微信在分享面板出现;选微信后能进对话选联系人;发出去对方能播放;微信不会过度压缩(对比原片) |
| **预估工时** | 1h(假设 H7 已经产出视频文件) |
| **失败兜底** | 微信不接受 content://(罕见)→ 改用 file:// + 关掉 FileUriExposedException 检查;微信过度压缩 → 主动降码率到 2Mbps 以下 |

### 🟡 H9 · Android WebView 内 PlayCanvas 性能 + SPZ loader 可用性

| 项 | 内容 |
|---|---|
| **完整假设** | (a) Android System WebView 里跑 PlayCanvas Engine,fps 仍 ≥ 60fps;**(b) `@spz-loader/playcanvas`(社区包 0.3.1)能在 Android WebView 里成功加载 .spz 文件并交给 PlayCanvas 渲染** |
| **为什么 H2 通过 ≠ H9 通过** | Magic7 Pro 上 Chrome ≠ Android System WebView,WebView 是分离的可更新组件,部分 GPU 路径、CPU/内存预算可能不同。同时 `@spz-loader` 是社区包,版本号偏早期(0.3.1),Android WebView 内未有公开实测案例 |
| **验证手段** | 1. 建最小 Android Activity + WebView<br/>2. WebView 加载本地 HTML + PlayCanvas Engine + `@spz-loader/playcanvas`<br/>3. 用 Scaniverse 扫一个测试场景,导出 .spz 放到 assets/<br/>4. WebView 加载 + `createGSplatEntityFromSpzUrlAsync` 加载 → 测渲染 fps + SPZ 加载耗时<br/>5. 与 H2 Chrome 数据对比 |
| **通过标准** | (a) fps ≥ 60(≥ 50 合格,< 50 失败);(b) SPZ 加载耗时 < 3s;(c) 视觉质量与官方 demo 等效 |
| **预估工时** | 3h(原 2h 加上 SPZ loader 集成 1h) |
| **失败兜底** | (a) WebView 比 Chrome 慢明显 → X5 内核(腾讯)替代;最坏退回 PWA(A 路线)<br/>(b) SPZ loader 跑不通 → **回退 v0.2 路径**:Scaniverse 导出 PLY → 电脑跑 SplatTransform 转 SOG → PlayCanvas 加载 SOG。这就是 03 v0.3 § 二 Layer 2 删除时保留的"回退方案" |

### ⏸️ H4 · 自扫场景(仍推迟 M1)

不变。M1 实施阶段补做。工具优先级:**Scaniverse**(免费,设备端,绕过 WorkOS,已装到 Magic7 Pro)> KIRI Pro > 其他。
