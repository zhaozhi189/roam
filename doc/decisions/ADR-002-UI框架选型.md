# ADR-002 · UI 框架选型(待决)

> 状态:🟡 Proposed
> 日期:2026-05-12
> 关联:[01-需求说明.md § 7 Q2](../01-需求说明.md)

## 背景 / 上下文

Roam Web 应用需要一个 UI 框架,负责:

- 登录页 / 上传页 / 我的场景列表
- 漫游页的 HUD(顶部分享按钮、右下退出按钮等)
- 移动端响应式

UI 部分**不渲染 3D**(3D 部分交给 PlayCanvas Engine 独立运行),所以 UI 框架的选择主要看 **包体积 / 上手速度 / 与 PlayCanvas 共存**。

## 决策

**待定**,需要在 M1 单场景本地 Demo 之后(动手前)拍板。

## 备选方案

| 选项 | 包大小 | 学习曲线 | 与 PlayCanvas 共存 | 倾向 |
|---|---|---|---|---|
| **React 19 + Vite** | gzip ~45KB | 极熟 | ✅ 好(PlayCanvas 单独挂 canvas DOM) | ⭐⭐ |
| **Solid.js** | gzip ~7KB | 简单(类 React) | ✅ 好 | ⭐⭐⭐ |
| **纯 TypeScript + DOM** | gzip ~0 | 简单(无框架) | ✅ 最好 | ⭐⭐ |
| **Vue 3** | gzip ~30KB | 熟 | ✅ 好 | — |
| **Svelte** | gzip ~10KB | 中 | ✅ 好 | — |
| **Next.js / Nuxt** | 大 | 较重 | 不需要 SSR | ❌ |

## 决策点(待评估)

1. **3D 应用 UI 通常很简单**(就一个 canvas + 几个按钮),是不是连框架都不用?
2. Solid 在 3D 场景下流行度高,**包小、性能好、写法接近 React** —— 看起来是最佳折中
3. 个人对 React 熟悉度最高 —— 选 React 上线最快,但包大

## 后果(各选项各自的)

### 选 React
- ✅ 我熟,周末就能跑通
- ❌ 包大,移动端冷启动稍慢
- ❌ 与 3D 包体的 PlayCanvas 加在一起,首屏 200KB+

### 选 Solid
- ✅ 包极小(7KB),移动端友好
- ✅ 性能好(细粒度响应)
- ❌ 我没用过,需要 1-2 天上手
- ❌ 生态比 React 小,UI 组件库选择少

### 选纯 TypeScript
- ✅ 无依赖,包最小
- ✅ 性能最好
- ❌ 手写表单 / 路由 / 状态管理,会更慢
- ❌ 加新页面成本高

## 何时回头看

- M1 跑通后,**拿一个 splat 文件 + 几个按钮的 Demo 走一遍**,实测三种方案的首屏速度与开发体验
- 选定后写入本 ADR 的"决策"部分,状态改 ✅ Accepted
